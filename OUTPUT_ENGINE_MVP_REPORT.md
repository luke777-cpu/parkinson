# 출력 변화 엔진 MVP 구현 보고서

- 구현 버전: 0.12.1 Stable Test
- 구현일: 2026-07-22
- 기준 문서: `MASTER_SPEC.md`, `CODEX_OUTPUT_ENGINE_MVP.md`, `CODEX_IMPLEMENTATION_PROMPT.md`

## 구현 범위

이번 버전은 ON/OFF 자동 판정이나 약효 해석을 수행하지 않는다. 사용자가 하루 첫 기준 출력을 선택한 뒤 실제 변화를 느낄 때마다 `+10` 또는 `-10`을 기록하여 하루 출력곡선을 만든다. 여기에 복약·이상운동증·동결·근긴장이상·넘어질 뻔 사건을 출력과 독립적으로 기록하는 빠른 입력 기능을 추가했다.

## 데이터 보호

- 기존 앱 데이터 키 `yakhyo_log_v1`과 기존 이벤트는 유지한다. 새 복약 기록에 생성 시각·당시 출력·입력 방식 메타데이터만 호환 필드로 추가한다.
- 출력 이벤트는 별도 키 `medicationDiary.outputEvents.v1`에 저장한다.
- 빠른 증상 사건은 별도 키 `medicationDiary.quickEvents.v1`에 저장한다.
- 출력 이벤트의 값은 0~100, 10점 단위로 검증한다.

## OutputEvent 모델

각 기록은 다음 정보를 보존한다.

- 고유 ID
- ISO 시각과 현지 날짜
- `anchor`, `delta`, `manual_correction`, `retrospective_output` 구분
- 이전 출력, 변화량, 새 출력
- 입력 방식
- 2시간 공백 후 계속 입력한 경우의 낮은 신뢰도
- 생성 및 수정 시각
- 소급 여부, 소급 입력 방식, 선택 메모

중간 기록을 수정하거나 삭제하면 이후 `delta` 이벤트는 원본 이벤트를 유지한 채 시간순으로 다시 계산한다. 뒤 기록이 있는 기준점을 삭제하면 기록 손상을 막기 위해 삭제를 차단하고 기준값 수정을 안내한다.

- 기준점 및 낮은 신뢰도 기록의 별도 마커

그래프 연결선은 관찰을 돕기 위한 선일 뿐, 기록 사이 실제 출력을 확정하지 않는다는 안내 문구를 표시한다.

## 0.10.1 브라우저 실행 오류 수정

- 출력 UI 전체가 ES 모듈 로딩 성공 뒤에만 생성되던 단일 실패 지점을 제거했다.
- 원본 모듈은 테스트 가능한 소스로 유지하고, 브라우저는 `js/output-ui.bundle.js` 단일 일반 스크립트를 실행한다.
- `file://` 직접 열기에서 발생할 수 있는 모듈 CORS 오류와, 하위 모듈 1개 누락이 전체 출력 UI를 비우는 문제를 차단했다.
- 로딩 전 상태를 정적 HTML에 표시하여 실행 파일이 배포되지 않은 경우에도 빈 영역 대신 로딩 상태가 보이도록 했다.
- 서비스워커 캐시를 `0.10.1`로 갱신하고 브라우저 실행 번들을 오프라인 자산에 추가했다.
- 검증 스크립트가 HTML이 참조하는 로컬 CSS·JS 파일 존재 여부와 브라우저 번들의 일반 스크립트 구문을 검사하도록 강화했다.
- 직접 파일 실행과 HTTP 실행에서 각각 기준 출력 `40 → 50 → 40`, 그래프 3개 점, 기본 닫힘 상태를 확인했으며 수집된 JavaScript 오류는 0개였다.
- HTTP 점검에서 `index.html`, CSS, 실행 번들, 원본 모듈 4개, `sw.js`, `manifest.json`이 모두 200으로 응답했고 CSS/JavaScript MIME 형식도 정상임을 확인했다.

## 0.11.0 지난 시간 출력 소급 기록

소급 기록은 변화량이 아닌 당시의 절대 출력값을 저장한다.

```json
{
  "eventType": "retrospective_output",
  "timestamp": "실제 상태가 있었던 시각",
  "createdAt": "나중에 입력한 시각",
  "previousOutput": 50,
  "delta": null,
  "newOutput": 70,
  "inputMethod": "backdated_manual",
  "retrospective": true,
  "note": "선택 메모"
}
```

재계산 규칙:

1. 모든 이벤트를 `timestamp`순으로 안정 정렬한다.
2. `anchor`, `manual_correction`, `retrospective_output`은 해당 시점의 새 절대 기준이 된다.
3. 그 뒤의 `delta`는 직전 출력부터 `+10/-10`을 순서대로 적용하며 `previousOutput`과 `newOutput`을 함께 갱신한다.
4. 다음 절대 기준 이벤트를 만나면 앞선 소급점의 영향은 종료된다.
5. 삽입·수정·삭제는 기존 이벤트 ID와 원시 `delta`를 보존한 채 전체를 다시 계산한다.
6. 소급점이 첫 기준이었다가 삭제된 경우 뒤의 원시 `delta.previousOutput`을 복구 기준으로 사용하여 기록을 잃지 않는다.

입력 가능 범위는 오늘을 포함한 최근 7일이며 미래 시각은 거부한다. 같은 분에 기존 기록이 있으면 기존 기록 수정, 새 기록 추가, 취소 선택창을 표시한다. 기존 출력 저장 키 `medicationDiary.outputEvents.v1`과 기존 앱 데이터 `yakhyo_log_v1`은 변경하지 않았다.

## 남은 확인 사항

- 실제 Android PWA 설치 환경에서 터치감과 화면 크기는 기기 수동 시험이 필요하다.
- 서비스워커의 실제 업데이트 안내와 완전 오프라인 실행은 설치형 브라우저에서 최종 확인이 필요하다.
- 실제 사용자에게 최소 하루 사용하게 한 뒤, 10점 변화 단위와 버튼 입력 빈도가 체감곡선에 적절한지 검토해야 한다.
- `assetlinks.json`의 Play App Signing SHA-256 지문은 실제 지문으로 교체해야 한다.

## 0.12.0 출력 중심 빠른 사건 기록

메인 화면 순서는 `현재 출력 → -10/+10 → 출력 그래프 → 빠른 사건 기록 → 지난 시간 출력 기록 → 상세 기록`으로 정리했다. 새 빠른 사건은 출력값을 바꾸지 않고 당시 출력의 스냅샷만 보존하며 ON/OFF 이벤트를 자동 생성하지 않는다.

### QuickEvent 모델과 저장 규칙

- 순간 사건: `freezing`, `near_fall`
  - `timestamp`, `createdAt`, `outputAtEvent`, `inputMethod: quick_event`, `retrospective: false`, 메모를 저장한다.
  - 동일 사건을 1초 안에 다시 누르면 두 번째 입력을 저장하지 않는다.
- 지속 사건: `dyskinesia`, `dystonia`
  - 첫 입력은 `status: open`과 `startTimestamp`, `startOutput`을 저장한다.
  - 같은 버튼의 다음 입력은 `status: closed`, `endTimestamp`, `endOutput`을 갱신한다.
  - 진행 중 상태는 새로고침 뒤에도 유지되며, 수정창에서 빠뜨린 종료 시각을 나중에 입력할 수 있다.
- 복약:
  - 기존 약 선택·용량·시간 입력창을 재사용한다.
  - `yakhyo_log_v1`의 복약 이벤트에 `createdAt`, `outputAtEvent`, `inputMethod`, `retrospective`를 추가한다.
- 빠른 증상 저장 키와 기존 출력·앱 저장 키는 서로 독립적이다.

### 표시와 편집

- 선택 날짜의 출력점과 복약·순간·지속 사건을 같은 그래프에 서로 다른 최소 마커로 표시한다.
- 통합 최근 목록은 실제 시각순으로 정렬하며 출력·복약·증상 사건을 수정하거나 삭제할 수 있다.
- 이상운동증과 근긴장이상 버튼은 진행 중일 때 시작 시각과 종료 안내를 표시한다.
- 지난 시간 출력은 메인 버튼과 출력 기록 관리 메뉴 양쪽에서 같은 안전한 입력창을 연다.

## 0.12.1 Stable Test 안정화

`WORK_ORDER_v0.9.md`의 배포 안정화 범위만 적용했으며 새 분석 알고리즘이나 데이터베이스 재설계는 하지 않았다.

### 수정 내용

- 기존 내부 상태값 `off`, `rising`, `on`, `falling`은 그대로 유지하면서 화면에서는 각각 낮은 출력 구간, 출력 상승 구간, 높은 출력 안정 구간, 출력 하강 구간의 **해석 메모**로 표시한다.
- 해석 메모를 저장해도 `medicationDiary.outputEvents.v1`의 현재 출력은 변경하지 않는다.
- 기존 `state:"rising" + riseResult` 구조를 그대로 읽어 `delayed_on`, `incomplete_on`, `on_failure` 표시용 사건으로 변환한다.
- Delayed ON, 불완전 ON, ON Failure를 메인 출력 그래프에 D/I/F 마커로 표시하고 통합 최근 목록에서 수정·삭제할 수 있게 했다.
- 새 방식의 불완전 ON이 과거 `state:"partial"` 시간만 집계하던 보고서에서 0으로 보이던 문제를 수정했다.
- 보고서와 오늘 요약에 Delayed ON, 불완전 ON, ON Failure 횟수를 각각 표시한다.
- 첫 복약부터 첫 ON 기록까지의 기존 수치는 확정적인 Delayed ON 판정으로 표시하지 않고 `첫 복약→첫 높은 출력 기록`으로 명확히 구분했다.
- 새 해석 메모에는 당시 주 출력의 스냅샷과 생성·수정 시각을 선택 필드로 보존한다. 과거 기록은 기존 필드 그대로 호환된다.
